# v4 모바일/접근성/QA 체크리스트

## 목적

이 문서는 v4 디자인이 실제 구현에서 깨지지 않도록 모바일, 접근성, 시각 QA 기준을 정리한다.

## 반응형 기준

| 구간 | 기준 | 레이아웃 |
| --- | ---: | --- |
| 데스크톱 | 1200px 이상 | 모바일 정보 순서를 유지한 2열 확장 |
| 태블릿 | 768~1199px | 주요 패널 1열 또는 2열 혼합 |
| 모바일 | 390~767px | 430px 기준 한 열, 하단 탭 |
| 소형 모바일 | 360~389px | 캘린더 금액 축약, 카드 1열 전환 가능 |

### 홈

2. 상태 요약

## 모바일 금지 사항

- 캘린더 셀 안에 긴 사용처명을 넣지 않는다.
- 금액이 셀 너비를 넘으면 `3.5만`처럼 축약한다.
- 하단 탭이 콘텐츠를 가리지 않도록 하단 패딩을 둔다.
- 가로 스크롤이 생기지 않게 한다.
- 버튼 텍스트를 한 줄에 억지로 넣지 않는다.
- 상단 액션을 3개 이상 넣지 않는다.
- 캘린더만 어두운 테마로 분리하지 않는다.
- AI 채팅 UI를 첫 화면 중심으로 두지 않는다.

## 접근성 체크리스트

### 색상

- 위험 상태는 색상과 텍스트 라벨을 함께 사용한다.
- 구독일은 앰버 색과 `구독` 또는 서비스명 라벨을 함께 사용한다.
- 텍스트 대비는 흰 카드와 연한 배경 위에서 충분해야 한다.

### 키보드

- 탭, 버튼, 날짜 셀은 키보드 포커스가 보여야 한다.
- 선택된 날짜는 `aria-selected` 또는 명확한 상태값을 갖는다.
- 아이콘 버튼은 `aria-label`을 갖는다.
- 하단 탭은 현재 위치를 `aria-current="page"`로 표시한다.

### 스크린리더

- 캘린더는 날짜와 지출 정보를 읽을 수 있어야 한다.
- 코치 카드에는 섹션 제목이 있어야 한다.
- 폼 입력에는 라벨이 연결되어야 한다.
- 금액 변화는 필요하면 상태 메시지로 알려야 한다.

### 폼 오류

- CSV 업로드 오류는 색상만으로 표시하지 않는다.
- 금액 입력 오류는 기대 형식을 문장으로 안내한다.
- 저장 실패 시 입력값을 유지한다.

## 시각 QA 체크리스트

### 첫인상

- 첫 화면에서 실제 앱처럼 보이는가?
- 서비스명과 이번 달 소비 금액이 상단에 명확히 보이는가?
- 소비 캘린더가 제품의 중심 기능처럼 보이는가?
- AI라는 단어보다 조정 문장과 미션이 먼저 보이는가?

### 정보 위계

- 이번 달 소비 금액이 가장 크게 보이는가?
- 목표 진행률이 숫자와 막대로 함께 보이는가?
- 오늘 가능예산, 저축 예상, 구독비가 같은 높이와 밀도로 정리되어 있는가?
- 구독 지출이 홈에서 숨지 않는가?

### 캘린더

- 안정, 구독, 초과, 선택 상태가 구분되는가?
- 모바일에서 날짜 셀의 금액과 상태 점이 겹치지 않는가?
- 선택일 상세가 캘린더 밖에 따로 보여 셀을 복잡하게 만들지 않는가?
- 범례 없이도 대략적인 상태를 이해할 수 있는가?

### 코치

- 문장이 사용자 탓처럼 들리지 않는가?
- 미션이 7일 안에 실행 가능한 행동인가?
- 예상 절약액이 구체적인가?
- 긴 분석문보다 행동 카드가 먼저 보이는가?

### 금융 신뢰감

- 금액 표기가 일관되는가?
- 입금은 민트, 지출은 코랄로 구분되는가?
- 실제 은행 연동이 없는 버튼을 과장해서 보여주지 않는가?
- 개인정보/금융 인증정보 미수집 안내가 필요한 화면에 있는가?

## 디자인 QA 완료 기준

- 모든 뷰포트에서 가로 스크롤이 없다.
- 버튼 높이는 최소 40px이다.
- 카드 radius는 8px 기준을 유지한다.
- 금액과 상태 라벨이 겹치지 않는다.
- 하단 탭이 콘텐츠를 가리지 않는다.
- 캘린더는 한 화면 안에서 앱의 핵심으로 보인다.
- 코치 기능은 결론, 이유, 미션, 예상 절약액을 포함한다.
