## 0. 평가 목적과 원칙

이 문서는 서로 목적이 다른 두 평가를 분리한다.

- **블랙박스 평가(공식 예상점수, 100점)**: 코드를 보지 않고, 서비스를 처음 접한 공모전 심사위원이 제출 URL에서 직접 사용한 경험만 평가한다. 공식 Track B 심사 항목과 배점을 그대로 사용한다.
- **화이트박스 평가(기술·배포 준비도, 100점)**: 코드, 테스트, 빌드, 배포 설정, 데이터 흐름, AI 연동 구조를 확인해 안정성과 확장 가능성을 평가한다.
- 두 점수는 합산하지 않는다. 블랙박스 점수는 공모전 심사 예상치이고, 화이트박스 점수는 그 결과를 뒷받침하는 기술 신뢰도다.
- 최종 보고에는 `블랙박스 __/100`, `화이트박스 __/100`, `배포 준비 판정`을 각각 기록한다.
- 확인하지 못한 항목은 추정해서 점수를 주지 않고 `NT(Not Tested)`로 표시한다. 공식 점수 추정이 필요하면 NT는 보수적으로 미충족 처리하고 신뢰도를 함께 적는다.

- [ ] 요청 시 제출할 수 있도록 소스코드와 실행·빌드 문서가 정리돼 있다.
- [ ] 제출 버전과 지원서 설명이 일치하며 미구현 기능을 구현된 것처럼 표현하지 않는다.

게이트 실패 시 총점과 무관하게 `제출 중단 후 수정`으로 판정한다.

### 증거 기록

모든 점수에는 최소 한 개의 증거를 붙인다.

- `관찰`: 화면 조작, 계산 결과, 오류 메시지, 테스트 출력처럼 직접 확인한 사실
- `추론`: 관찰을 근거로 한 평가자의 해석. 반드시 추론이라고 표시
- `NT`: 시간·권한·호출 제한 때문에 확인하지 못함
- 동일한 결함을 여러 세부 항목에서 중복 감점하지 않는다. 가장 직접적인 항목에서 감점하고 다른 항목에는 교차 참조만 남긴다.

## 4. 블랙박스 표준 시나리오

### 사전 조건

- 새 브라우저 프로필 또는 저장소가 비어 있는 컨텍스트를 사용한다.
- 코드, README, 내부 테스트 결과를 보지 않는다.
- PC 1440×900 전후와 모바일 390×844를 각각 확인한다.
- AI 관련 사용자 조작은 정상 분석 1회를 원칙으로 하며, 전체 실제 호출 한도 5회를 넘지 않는다.
- 각 단계에서 소요 시간, 클릭 수, 막힌 지점, 화면 문구를 기록한다.

### 심사위원 첫 사용 흐름

1. **0~30초: 첫 진입**
   - URL 접속, 첫 화면 로딩, 서비스 목적, 첫 행동, 빈 상태를 확인한다.
   - 설치 기능이 보일 때만 설치·실행 경험을 추가 확인한다.
2. **30~90초: 샘플 또는 첫 데이터**
   - 화면에 보이는 경로로 샘플 데이터를 불러오거나 소비 한 건을 입력한다.
   - 성공 피드백과 캘린더·요약·최근 내역 반영을 확인한다.
3. **90~150초: 목표 설정과 수정**
   - 월수입, 월 소비 목표, 저축 목표, 집중 카테고리를 수정한다.
   - 현실적인 값과 비현실적인 값을 각각 넣어 경고와 상단 상태 문구의 일관성을 본다.
4. **150~220초: 소비 관리와 캘린더**
   - 소비 추가·수정·삭제, 날짜 선택, 필터, 정기결제, 빈 달 또는 빈 날짜 상태를 확인한다.
   - 금액·건수·진행률이 화면마다 같은지 비교한다.
5. **220~300초: AI 코치**
   - AI 화면 진입 후 분석을 한 번 실행한다.
   - 로딩 문구, 대기 시간, 중복 호출로 보이는 변화, 결과의 구체성, 실패·폴백 문구를 확인한다.
6. **300~360초: 설정과 데이터 이동**
   - 알림 내역, 신뢰 안내, CSV 내보내기·가져오기, 잘못된 날짜·금액 처리, 초기화를 확인한다.
7. **모바일 및 재방문**
   - 같은 핵심 흐름을 모바일에서 축약 확인한다.
   - 같은 프로필로 새로고침·재방문해 저장 상태와 복구 경험을 확인한다.

## 6. 화이트박스 표준 검증 절차

1. **구조 파악**: 앱 진입점, 상태 저장소, 도메인 계산, AI API, CSV·알림·배포 경계를 그린다.
2. **위험 중심 코드 검토**: 목표 계산, 거래 CRUD, 날짜 처리, 자동 분류, 캐시·중복 호출, 타임아웃·폴백을 우선 확인한다.
3. **정적·자동 검증**: 저장소의 공식 verify 명령을 실행하고 타입, 린트, 단위·통합·브라우저 테스트, 프로덕션 빌드 결과를 기록한다.
4. **배포 검증**: 제출 URL과 현재 소스 버전의 일치 여부, 직접 경로, 새로고침, 환경변수, 백업 배포를 확인한다.
5. **안전성 점검**: 비밀키 노출, 입력 검증, localStorage 데이터, 로그 개인정보, 의존성·라이선스를 확인한다.
6. **확장 시뮬레이션**: 실제 거래 공급자 하나 또는 계정 동기화 하나를 추가한다고 가정해 수정 범위와 결합도를 평가한다.
7. **AI 호출 제한 점검**: 실제 호출은 최대 5회로 제한하고 나머지는 mock·테스트·코드 증거로 확인한다.

## 10. 최종 해석 기준

### 블랙박스 공식 예상점수

- 90~100: 수상 경쟁력이 높은 완성도. 치명적 첫인상 결함이 없어야 한다.
- 84~89: 제출 경쟁력은 충분하나 심사 중 드러날 P1·P2를 우선 제거해야 한다.
- 75~83: 핵심 아이디어는 전달되지만 완성도 또는 사용성에서 감점 가능성이 크다.
- 74 이하: 제출 전 핵심 흐름과 가치 전달을 다시 정비해야 한다.

### 화이트박스 기술·배포 준비도

- 90~100: 작은 기능 확장 후 실제 공개 운영을 고려할 수 있다.
- 85~89: 공개 베타 게이트 충족 여부를 확인하고 제한된 운영이 가능하다.
- 75~84: 공모전 데모는 가능하지만 실제 사용자 확대 전 보완이 필요하다.
- 65~74: 구조·안정성 부채가 확장 속도를 크게 낮출 가능성이 있다.
- 64 이하: 기능 추가보다 핵심 구조와 장애 대응을 먼저 재정비한다.

최종 제출 판단은 `블랙박스 점수`, `화이트박스 점수`, `참가 적격성 게이트`, `P0/P1 결함` 네 가지를 함께 보고 내린다.
